# Folk

**Быстрый и надёжный PHP application server — народный по духу, написанный ИИ.**

Folk запускает ваше PHP-приложение как долгоживущий процесс, а не поднимает его с нуля на каждый запрос. Это рантайм на **Rust**, встроенный как нативное **PHP-расширение**: между сервером и вашим кодом нет отдельного процесса с каким-то протоколом — PHP исполняется *внутри процесса*, рядом с рантаймом.

Цель простая: **скорость и надёжность без плясок с бубном** вокруг установки и настройки сервера приложений.

Folk требует **PHP, собранный с ZTS** (Zend Thread Safety). Дальше — два шага стандартными инструментами:

```bash
# 1. Установить расширение Folk — пред-собранный бинарник, Rust-тулчейн не нужен
pie install folk-project/ext-folk

# 2. Добавить адаптер под ваш фреймворк
composer require folk/laravel   # или folk/symfony, folk/spiral, folk/yii3
```

[PIE](https://github.com/php/pie) — установщик расширений от PHP Foundation; Composer подтягивает адаптер. Никаких самописных пулов воркеров, никакого склеивания протоколов.

---

## Для кого Folk

PHP-разработчики, которым нужна реальная производительность, но **без**:

- перехода на Go или другой язык,
- ручного управления пулами воркеров и event loop'ами,
- хрупкого ритуала «установи и настрой» каждый раз.

Если у вас приложение на Laravel / Symfony / Spiral / Yii3 и вы хотите, чтобы оно работало быстро и стабильно — Folk для вас.

## Чем Folk отличается

Большинство инструментов «ускорь PHP» запускают отдельный серверный процесс, который общается с вашими PHP-воркерами по сокету или пайпу через какой-то протокол. Это добавляет лишние детали, накладные расходы на сериализацию и второй процесс, который надо настраивать и поддерживать живым.

Folk идёт другим путём:

- **In-process, через PHP-расширение.** Rust-рантайм и ваш PHP-код живут в одном процессе. Между сервером и воркером нет межпроцессного протокола (мы осознанно отказались от подходов pipe / fork / MessagePack).
- **Rust под капотом.** Безопасность памяти и предсказуемая производительность там, где это важно — рантайм, пул воркеров, I/O.
- **Framework-agnostic.** Полноценные адаптеры для Laravel, Symfony, Spiral и Yii3.
- **Дистрибуция, которая просто работает.** Rust-рантайм поставляется как пред-собранное расширение через PIE — на целевой машине Rust-тулчейн не нужен; адаптеры — на Packagist.

## Чем Folk **не** является

Чтобы сразу выставить ожидания честно:

- **Не AOT-компилятор.** Ваш PHP по-прежнему исполняет Zend VM — Folk не компилирует PHP в нативный код.
- **Не замена PHP.** Это рантайм *для* PHP.
- **Не фреймворк.** Он запускает ваш существующий фреймворк, а не заменяет его.

## Народный по духу, написанный ИИ

Это то, что делает Folk необычным, и мы лучше скажем об этом сразу, чем спрячем.

**Folk разрабатывается ИИ-агентами под ревью человека по spec-driven процессу.** Контрибьюторы обычно не открывают pull request с кодом. Они приносят **хорошо сформулированную идею или спецификацию** — а реализацию по этой спеке производит ИИ.

Это значит, что самый ценный вклад здесь — **ясное мышление**. Чёткая постановка проблемы, проработанный дизайн, точная спека — вот что двигает проект. Код вторичен по отношению к хорошей спеке.

Почему мы работаем так:

- Это держит workspace из 14 репозиториев согласованным — каждое изменение проходит один и тот же цикл спека → research → реализация → проверка.
- Это снижает порог для *осмысленного* вклада: не нужно быть экспертом по Rust, чтобы влиять на Folk — нужна хорошая идея, выраженная ясно.
- Это честно. Код пишет ИИ; люди решают, что стоит делать, и ревьюят то, что выходит в релиз.

> **Folk** — от *folk*: народное, общими мозгами. Продукт, собранный из идей сообщества и воплощённый в код с помощью ИИ. *Быстрый PHP, сделанный общими силами.*

## Как контрибьютить

Жизнь идеи:

```
идея / issue
   → обсуждение (та ли это проблема? та ли форма решения?)
   → спека + research.md (что, зачем, prior art, trade-offs)
   → реализация ИИ (под ревью)
   → quality gates (см. ниже)
   → релиз
```

**Приносите идею или спеку, а не (обязательно) патч.** Откройте issue в трекере [`folk-releases`](https://github.com/Folk-Project/folk-releases). Чем яснее и проработаннее — тем быстрее она станет реальностью:

- **Сформулируйте проблему** до решения.
- **Покажите prior art** — как это решают RoadRunner / Swoole / FrankenPHP и другие?
- **Назовите trade-offs**, которые видите.
- **Будьте конкретны** — примеры, фрагменты конфига, ожидаемое поведение.

Хорошая идея, описанная так, чтобы ИИ-агент мог её реализовать без догадок, — самое полезное, что вы можете дать Folk.

## Что попадает в релиз

Изменение выходит только после прохождения ворот:

- **Rust:** `fmt` + `clippy` + тесты, всё зелёное.
- **End-to-end:** Docker smoke-тест против реального приложения на фреймворке должен пройти до любой публикации.
- **Ревью:** мейнтейнер принимает финальное решение о scope и о том, что выходит.

Ничего не попадает на crates.io или Packagist без прохождения end-to-end smoke-теста.

## Roadmap

Куда движется Folk (чтобы ваши идеи целились в ту же точку):

- **Embedded mode** — самодостаточный однофайловый бинарь: Rust-хост + статически слинкованный ZTS PHP + ваше приложение, для Go-style деплоя.
- **Concurrency** — конкурентность на воркер на базе Fibers / TrueAsync и стриминг ответов.

## Репозитории

Folk — это workspace из сфокусированных репозиториев:

| Репозиторий | Что это |
|---|---|
| `folk-core` / `folk-ext` | Rust-рантайм и PHP-расширение |
| `folk-api` | Plugin trait / API расширения |
| `folk-plugin-*` | Плагины HTTP, gRPC, jobs, metrics, process |
| `folk-builder` | Собирает `cdylib`-расширение под ваше приложение |
| `folk-sdk` | PHP SDK (`folk/sdk`) |
| `folk-laravel` / `folk-symfony` / `folk-spiral` / `folk-yii3` | Адаптеры под фреймворки |
| `folk-releases` | Docs, CI, пред-собранные бинарники, **трекер issue** |

## Лицензия

- **Rust-крейты** (`folk-core`, `folk-ext`, плагины, `folk-builder`, …): **Apache-2.0** — включая явный патентный грант.
- **PHP-пакеты** (`folk/sdk`, `folk/laravel` и остальные адаптеры): **MIT**.
